# PRD：Admin 模块（后台管理）

## 1. 模块定位

内部运营与运维使用，管理用户、积分、事件日志、系统配置。

**所属业务链路**：独立支撑层，不在用户主流程中。

## 2. 目标用户 & 场景

| 角色 | 场景 | JTBD |
|------|------|------|
| 产品运营 | 给新用户发试用积分 | 手工调整 |
| 研发 | 定位线上问题 | 查审计日志 |

## 3. 用户故事

- 作为 admin，我想查看/调整任意用户的积分余额。
- 作为 admin，我想查看事件日志、定位异常生成。
- 作为 admin，我想重置某用户的 onboarding 状态。

## 4. 功能需求

### 4.1 必备
- F1. `/admin/*` 强制 admin 身份（`adminAuth` 校验）。
- F2. 用户列表 + 详情 + 积分调整。
- F3. 事件日志筛选（按用户 / 类型 / 时间）。
- F4. 数据变更操作统一经 `AdminMutationBanner` 二次确认。

### 4.2 增强
- E1. 成本统计仪表盘。
- E2. 批量操作。

### 4.3 非目标
- NG1. 不做终端用户自助（见 Account 设置）。

## 5. 验收标准

- [ ] AC1. 非 admin 访问 `/admin/*` 返回 403
- [ ] AC2. 所有变更操作记入 eventLog
- [ ] AC3. 积分调整有 audit trail（谁 / 何时 / 变动额）

## 6. 关键指标

| 指标 | 目标 |
|------|------|
| 非法访问拦截率 | 100% |
| 操作审计完整性 | 100% |

## 7. 依赖 & 约束

- 依赖：`adminAuth`, `creditLedger`, `eventLog`
- 约束：生产环境 admin 列表通过环境变量或 DB 表维护

## 8. 风险

| 风险 | 应对 |
|------|------|
| admin 越权 | `adminAuth` 双层校验（middleware + 路由内） |
| 误操作 | `AdminMutationBanner` 二次确认 |
