## 为什么是 C12 持久化桥（VFM / ADL）

候选池里 C12 在 cycle 2 末尾还没出现 —— 它是 xu 那句「历史聊天对话存在哪里？」当场点出来的洞。我立刻 `grep -rn` 了 `storage.ts`，确认它的所有导出函数**零调用方**。意味着：每次刷新浏览器，所有项目、消息、生成任务、资产，全部蒸发。这件事的严重性高于候选池里任何一项。

| 候选 | VFM | 选 / 不选 |
|------|-----|----------|
| **C12** 持久化桥 | **96** | ✅ 本 cycle |
| C7 spec auto-pick | 80 | 不选：纯增量功能，丢数据这件事比它严重一个量级 |
| C13 base64 资源去重 | 72 | 不选：是 C12 的优化项，必须 C12 先到位 |
| C9 webm probe | 50 | 不选 |

C12 的得分构成：fail-reduction 极高（直接终结"刷新即丢"这一类用户事故）、reusability 高（store mutation 全部走同一条桥）、stability 高（atomic write + schema version + path whitelist 三层保护）、explainability 高（plain JSON，xu 可以 `cat` 直接读）、scalability 中（plain JSON 在 1k 项目级别完全够用，过了再迁 SQLite）、novelty 低（这就是个标准的桥，不耍花枪）。**ADL 排序里前四项全部满分，所以总分压过其他所有候选。**

## 回滚

- **持久化路径污染**：`rm -rf data/projects` 即可回到空状态；schema version 字段保证未来版本不会读旧文件出错。
- **debounce 太慢看起来像丢失**：把 `PERSIST_DEBOUNCE_MS` 从 250 调到 0（同步触发）；或把 8 个 `_schedulePersist(id)` 调用全部删掉，store 立刻回到 cycle 2 内存模式，0 副作用。
- **REST 滥用**：三条 route 都是文件级 idempotent，`rm -rf data/projects` 然后重启即可。
- **hydrate 意外覆盖用户输入**：hydrate 已经是"memory wins"策略 —— 只把磁盘上不在内存里的项目合进来；如果仍有问题，把 `useEffect` 里的 `store.hydrate()` 调用注释掉即可，磁盘仍在持续写入只是不在加载时回填。

每一步都设计成可独立回退。
