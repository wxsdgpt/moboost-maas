## 为什么是 spec validator（VFM / ADL 推理）

5 个候选打分后，K1 Spec Validator 拿到 **84 / 100**（其他依次 80 / 59 / 58 / 48）。决定因素：
- **频次**：每次上传、每次产出都跑，全 Stage 1-4 复用 → ×3 加权满分
- **失败减少**：把"用户提交一张方图想发 IG Reel" 这种错误从 Stage 4 烧掉模型 token 才发现，提前到 Stage 1 上传瞬间就出 blocker
- **用户负担**：xu 不需要记住"TikTok 9:16 是 1080×1920、cap 500 MB、最佳 9-15 秒"——validator 自己说
- **ADL 完美**：纯函数，无 LLM、无 I/O、无随机、无时间 → same input → same output forever。可以放心做 CI gate。

明确**没**做 K2 (spec recommender) 的原因：它需要启发式 + LLM，稳定性 / 可解释性都低一档。下个 cycle 再做。

## 回滚

万一发现问题：
- 校验逻辑不对：把 specValidator.ts 内部全部短路成 `return finalizeReport(spec, [], opts)`，validator 退化成 always-OK，即可从所有调用方安全降级
- deprecated wrapper 让老调用方崩溃：把 `assetSpecs.ts` 里的 `validateAsset` 字面恢复成原 4-check stub
- inspector 页坏：直接删除 `src/app/specs/inspector/page.tsx`，REST + lib 不受影响

每一步都设计成可独立回退的。
