## 为什么是 video probe 而不是新主题（VFM / ADL）

cycle 1 收尾时候选池里写得很清楚：
> **C5** 视频 mp4 atom box 时长探测 — 70 分 — 解锁视频上传立即出 blocker，validator 才能发挥全部价值

cycle 2 重新跑了一遍 VFM 把 C5 评到 **81 / 100**（高于 cycle 1 估的 70，原因是 cycle 1 完成后 K1 的下游效用变得明显 → fail-reduction 权重更高了）。其它候选：

| 候选 | 总分 | 为何不选 |
|------|------|----------|
| C7 spec auto-pick | 80 | 性质完全不同（启发式 + LLM）；切换主题违反"单个功能做到极致"原则 |
| C5 mp4 atom box ✅ | **81** | 纯函数，零依赖，纯字节解析，是 cycle 1 的直接补完 |
| C8 tsconfig target | 24 | trivial，独立 5 分钟 cycle 即可 |
| C6 视觉 safe-zone | 49 | <50 不立项 |

**ADL 的关键考量**：cycle 1 立的原则是「validator 必须是纯函数 / 零 LLM / 零 I/O / 零随机」。如果引入 mp4box.js 或 fluent-ffmpeg 来做视频探测，就会污染整条上传管线 —— 每次上传都跑一段 npm 包代码，还可能 shell-out 到 ffmpeg，违反原则。手写 atom box reader 是让 cycle 1 的纯函数承诺继续传递到视频侧的唯一方法。
